Nacos

Nacos 是服务注册中心和配置中心，是一个基于云原生架构的动态服务治理和配置管理平台。

  • 服务发现及注册：可以服务注册到 Nacos 注册中心，并通过 Nacos 实现服务的自动发现和负载均衡，从而实现服务的良好可用性和弹性伸缩
  • 服务治理：可以通过 Nacos 实现服务的健康检查、故障转移、服务限流、熔断降级等治理能力，从而提高服务的可靠性和稳定性
  • 配置管理：可以将应用程序的配置信息存储在 Nacos 的配置中心，通过 Nacos 实现动态配置管理和灰度发布，从而实现应用程序的动态调整和部署
  • 事件监听和推送：可以通过 Nacos 实现配置变更、服务注册和注销等事件的监听和推送，从而实现应用程序的自动化部署和管理

AP+CP

Nacos 支持 AP 和 CP 两种模式，这样设计是因为 Nacos 的主要应用是用作注册中心和配置中心。对于注册中心来说，需要提供高可用性，否则会影响所有服务的互相调用。对于配置中心来说，需要提供高一致性，不同的机器接收到的配置不一致是不能接受的。可以根据具体的使用场景来修改配置文件，从而进行选择，默认是 AP。

在金融、支付等对数据一致性要求非常高的业务场景下可以选择 CP 模式，那么当发生网络分区或故障时，为了保证数据一致性，Nacos 会自动对服务进行隔离和恢复，同时导致部分服务不可用。在网站、游戏等对可用性要求更高的场景下可以选择 AP 模式，那么当发生网络分区或故障时，Nacos 会优先保证服务的可用性，在这个前提下再去尽可能保证数据一致性。

配置变化

服务端所发起的和客户端之间的长连接（保持一定的心跳机制）可以令配置变化具备良好的实时性，但是服务端需要维护大量的 TCP 连接，这会占用大量的内存和 CPU 资源，也容易受到网络抖动等因素的影响。客户端主动轮询的实现比较简单，但无法保证数据变化的实时性，客户端不断轮询也会对服务端造成压力。

Nacos 1.x 采用综合了长连接和轮询的长轮询，客户端发起轮询，但是 Nacos 不立即返回，而是将请求挂起，在一段时间内保持着一个有效连接，超时或者存在配置变化再返回响应，然后客户端再发起一次轮询。长轮询可以综合轮询和长连接的优点，缺点是需要频繁地发起 HTTP 请求，会增加网络开销，并受到网络延迟等因素的影响，导致实时性不如长连接。

Nacos 2.0 中采用 gRPC 和 Rsocket 实现了长连接 RPC 调用和推送来减少服务节点间的 HTTP 通信压力，同时还新增一个链接层来将不同类型不同来源的 Request 请求转化为相同语意的功能数据结果，复用业务处理逻辑，此时流量控制和负载均衡等功能也会在链接层处理。
